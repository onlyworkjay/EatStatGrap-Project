1. 프로젝트 개요

 서비스명
EatStatGram

서비스 정의
탐색부터 비교, 기록, 공유까지 햄버거 선택에 필요한 정보를 연결하는 메뉴 비교 플랫폼

프로젝트 소개
EatStatGram은 사용자가 햄버거 메뉴를 검색하고 가격, 칼로리, 단백질, 후기 정보를 비교하여 더 쉽게 메뉴를 선택할 수 있도록 돕는 서비스이다.

사용자는 브랜드명 또는 메뉴명을 기준으로 메뉴 정보를 조회하고, 여러 메뉴를 비교한 뒤 **“이걸로 정했어!”** 기능으로 자신의 선택을 기록할 수 있다.

선택 이후에는 간단한 후기를 작성하여 다른 사용자와 경험을 공유할 수 있다.

핵심 가치
- 파편화된 메뉴 정보 통합
- 가격, 영양, 후기 기반의 합리적 메뉴 선택 지원
- 사용자 선택 기록과 후기 기반의 메뉴 트렌드 제공
- 관리자와 사용자 제보를 통한 데이터 최신성 관리

2. 핵심 문제 정의

사용자는 메뉴를 고르기 전 다음과 같은 불편함을 겪는다.

- 브랜드별 메뉴 가격과 영양정보가 흩어져 있다.
- 비슷한 메뉴를 한눈에 비교하기 어렵다.
- 실제 사용자의 최근 후기와 선택 흐름을 확인하기 어렵다.
- 메뉴 정보를 확인하기 위해 공식 홈페이지, 포털 검색, 블로그 등을 오가야 한다.

3. 핵심 서비스 흐름

검색한다→ 비교한다 → 고른다→ 기록한다→ 공유한다

9. EAT 메뉴 탐색

EAT 페이지는 검색 중심이 아닌, 이미지 중심의 메뉴판 형태로 구성한다.

사용자는 브랜드와 메뉴 카드를 클릭하며 가볍게 메뉴를 탐색하고, STAT 비교 기능 또는 GRAM 후기 기능으로 이동할 수 있다.

EAT 메인 구성
- 신제품 배너
- 전체 메뉴 인기순 목록
- 브랜드 버튼 목록

브랜드 예시:
- 전체
- 맥도날드
- 버거킹
- 롯데리아
- 맘스터치
- 노브랜드

메뉴 카드 정보
- 메뉴 이미지
- 메뉴명
- 브랜드명
- 가격
- 칼로리
- 비교함 담기

메뉴 상세 카드
메뉴 카드를 클릭하면 카드를 확장해서 추가로 상세정보를 제공한다.

상세 정보:
- 영양정보 상세
- 알러지 정보
- 정보 제공자
- 최근 업데이트 날짜
- 후기 보러가기

조회 정책
최초 진입 시 전체 메뉴를 모두 조회하지 않고, 인기 메뉴 기준으로 일부 메뉴만 우선 조회한다.

목적:
- 초기 로딩 속도 개선
- 모바일 환경 최적화
- 사용자 탐색 집중도 향상

추가 메뉴는 브랜드 선택 또는 더보기 버튼으로 조회한다.





10. STAT 검색 및 비교

STAT 페이지는 빠른 검색, 정렬, 비교에 집중한 핵심 기능 페이지이다.

검색 조건
- 브랜드 선택
- 메뉴명 검색
- 정렬 기준 선택

정렬 기준
- 가격 낮은순
- 가격 높은순
- 칼로리 낮은순
- 칼로리 높은순
- 단백질 낮은순
- 단백질 높은순
- 인기순

검색 결과 표시 정보
STAT 검색 결과는 빠른 비교를 위해 이미지 없이 텍스트 중심으로 제공한다.

- 브랜드명
- 메뉴명
- 가격
- 칼로리
- 단백질
- 비교함 담기 버튼

비교함 정책
- Zustand를 사용하여 비교함 상태를 관리한다.
- 페이지 이동 중에도 비교함을 유지한다.
- 비회원은 최대 2개, 회원은 최대 3개까지 담을 수 있다.
- 같은 메뉴는 중복으로 담을 수 없다.
- 삭제 기능을  제공

비교 결과 표시 정보
- 브랜드명 작은 메뉴 이미지 또는 브랜드 이미지
- 메뉴명
- 가격
- 칼로리
- 단백질

선택 완료 화면
사용자가 **“이걸로 정했어!”** 버튼을 클릭하면 선택 기록을 저장하고 선택 완료 화면으로 이동한다.

표시 문구:
 - 선택 완료!  
 - 오늘의 메뉴가 결정되었습니다.  
 - 이 선택은 나중에 다시 확인할 수 있어요.  
 - 최근 이 메뉴를 선택한 사용자들의 후기를 확인해보세요.

제공 버튼:
- Gram 작성하기
- 다른 Gram 보러가기
- 비교 계속하기










11. “이걸로 정했어!” 기능

기능 설명
로그인한 사용자는 비교 페이지에서 최종 선택한 메뉴를 기록할 수 있다.

저장 정보
- 사용자
- 선택한 브랜드
- 선택한 메뉴
- 선택 시간

정책
- 선택 기록은 인기 메뉴 집계와 개인 선택 기록에 활용한다.
- 선택 이후 2시간 이내에 해당 메뉴 후기를 작성할 수 있다.
- 중복 저장 방지를 위해 사용자, 메뉴, 선택 시간을 기준으로 중복 처리를 고려한다.
- 하루 최대 3개 기록저장

12. GRAM 후기 기능

GRAM은 사용자가 선택한 메뉴에 대해 사진과 간단한 후기를 공유하는 카드형 피드이다.

작성 조건
- 로그인한 회원만 작성 가능
- “이걸로 정했어!” 선택 이후 2시간 이내 작성 가능
- 사진과 간단한 후기 작성 가능
- 하루 동일메뉴 1건, 전체 3건 등록 가능

게시 기간 정책
- 작성 직후 상태는 ACTIVE
- 작성 후 7일이 지나면 EXPIRED 상태로 변경
- GRAM 메인 피드에서는 ACTIVE 게시글만 조회
- 메뉴 상세에서는 해당 메뉴의 ACTIVE / EXPIRED 후기를 일부 조회 가능
- 마이페이지에서는 본인의 ACTIVE / EXPIRED 후기를 조회 가능
- 관리자는 전체 상태를 조회 가능

후기 상태값
- ACTIVE : 노출 중
- EXPIRED : 게시 기간 만료
- USER_DELETED : 작성자 삭제
- HIDDEN : 관리자 숨김 처리
- DELETED : ADMIN 삭제 처리

후기 표시 정보
- 메뉴 이미지 또는 사용자가 업로드한 사진
- 메뉴명
- 브랜드명
- 작성자 닉네임
- 후기 내용
- 좋아요 수
- 댓글 수
- 작성 시간
- 수정 시간

비회원의 경우 사용 불가하다.

- 닉네임
- 가입일

- 작성 후기 수
- 최근 후기 3건



14. 좋아요 / 댓글

좋아요 정책
- 좋아요는 토글 방식으로 동작한다.
- 동일 사용자는 동일 게시글에 중복 좋아요를 할 수 없다.
- 좋아요 수는 쩝쩝박사 랭킹에 활용한다.

댓글 정책
- 로그인한 회원만 작성 가능하다.
- 작성자만 수정 가능하다.
- 댓글 삭제는 soft delete 방식으로 처리한다.
- 삭제된 댓글은 “삭제된 댓글입니다.”로 표시한다.

15. 쩝쩝박사 랭킹

최근 7일간 GRAM 후기에서 받은 좋아요 수를 기준으로 TOP 5 사용자를 표시한다.

표시 방식
- 1등 : 금메달 뱃지
- 2등 : 은메달 뱃지
- 3등 : 동메달 뱃지
- 4~10등 : 쩝쩝박사 아이콘

17. 정보 수정 제보

사용자는 메뉴 정보가 잘못되었거나 업데이트가 필요하다고 판단될 경우 정보 수정 제보를 할 수 있다.

입력 방식
자유 게시글이 아닌 폼 형태로 입력받는다.

제보 가능 항목
- 가격
- 칼로리
- 단백질
- 판매 여부
- 기타 의견

처리 흐름
사용자 제보→ 관리자 검토→ 승인 시 메뉴 정보 업데이트 → 반려 처리

제보 상태값
- PENDING : 등록 및 검토 대기
- ACCEPTED : 승인
- REJECTED : 반려

마이페이지
사용자는 본인이 작성한 제보 내역과 처리 상태를 확인할 수 있다.




18. 신고 기능

신고 대상
- GRAM 후기
- 댓글
- 사용자

신고 사유
- 욕설 / 비방
- 광고 / 도배
- 허위 정보
- 부적절한 이미지
- 개인정보 노출
- 기타

신고 정책

- 관리자가 검토 후 숨김, 반려, 조치 완료 상태로 처리한다.

신고 상태값

- REVIEWING : 검토 중
- ACCEPTED : 신고 인정
- REJECTED : 신고 반려
- RESOLVED : 처리 완료

21. 이미지 업로드 정책

적용 대상
- GRAM 후기 이미지
- 메뉴 대표 이미지

정책
- 이미지는 Amazon S3에 저장한다.
- 이미지 용량과 로딩 속도를 고려하여 리사이징을 적용한다.
- Thumbnailator 라이브러리 사용을 우선 검토한다.
- 최대 800px 또는 1200px 기준으로 리사이징한다.
- 가로세로 비율은 유지한다.

예외 처리
이미지 업로드 성공 후 DB 저장에 실패하면 업로드된 S3 객체를 삭제한다.

흐름:
S3 이미지 업로드→ DB 저장 시도→ DB 저장 실패→ S3 객체 삭제

22. 데이터 관리 정책

기본 정책
- 크롤링은 사용하지 않는다.
- 메뉴 정보는 관리자가 직접 관리한다.
- v1은 햄버거 카테고리만 구현한다.
- 카테고리 구조는 향후 피자, 치킨, 샌드위치 등으로 확장 가능하게 설계한다.

콜드 스타트 방지
초기 데이터 부족 문제를 방지하기 위해 햄버거 프랜차이즈 5종의 대표 메뉴 데이터를 우선 확보한다.

예시:
- 맥도날드
- 버거킹
- 롯데리아
- 맘스터치
- KFC

23. Validation 정책

검증 항목
- 가격은 0원 이상이어야 한다.
- 칼로리는 0 이상이어야 한다.
- 단백질은 0 이상이어야 한다.
- 메뉴명은 공백일 수 없다.
- 브랜드명은 공백일 수 없다.

목적
- 잘못된 데이터 저장 방지
- 데이터 정합성 확보
- 프론트엔드 조작에 대한 서버 측 방어

24. API 문서화 / 로그 정책

API 문서화
Swagger 또는 SpringDoc을 활용하여 API 문서를 제공한다.

목적:
- API 요청 / 응답 형식 공유
- 프론트엔드 연동 효율 향상
- API 테스트 편의성 확보

로그 정책
Logback을 활용하여 주요 이벤트와 오류를 기록한다.

기록 대상:
- 로그인 실패
- 이미지 업로드 실패
- 메뉴 등록 실패
- 신고 처리 실패
- S3 객체 삭제 실패
- 관리자 처리 이력
